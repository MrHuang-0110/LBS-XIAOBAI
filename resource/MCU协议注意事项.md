# MCU 端 UART 协议对接速查

- 配套协议：[语音芯片交互协议.md](语音芯片交互协议.md) v0.7
- 目标 MCU：PY32F030K28U6TR
- 对端：ASRPRO 语音芯片
- 物理层：UART 9600 8N1 全双工

---

## 一、发送端（MCU → ASRPRO）

| 用途 | 帧文本 | 实际字节 | 字节数 | C 代码写法 |
|------|--------|----------|:--:|-----------|
| 播报语音 | `play=<id>\n` | `70 6C 61 79 3D <id_ascii> 0A` | 7-9 | `snprintf(buf, sz, "play=%u\n", id);` |
| 停止播报 | `stop\n` | `73 74 6F 70 0A` | 5 | `"stop\n"` |
| 心跳探测 | `ping\n` | `70 69 6E 67 0A` | 5 | `"ping\n"` |

**发送 API**：`HAL_UART_Transmit(&huart1, (uint8_t *)buf, len, 100);`

---

## 二、接收端（ASRPRO → MCU）

| 帧文本 | 语义 | 触发时机 | 解析要点 |
|--------|------|----------|----------|
| `wake\r\n` | 唤醒词识别 | 用户说"小白小白" | 精确匹配 `wake`（去帧尾后） |
| `cmd=<id>\r\n` | 命令词识别 | 用户说命令词 | 提取 id ∈ [1,16]（见协议第四节） |
| `done=<id>\r\n` | 播报完成通知 | 上次 `play=X` 播完 | 提取 id ∈ [17,41]，匹配 X |

**帧尾**：`\r\n`（0x0D 0x0A），最长 32 字节含帧尾。

---

## 三、接收解析流程

| 步骤 | 动作 | 说明 |
|:--:|------|------|
| 1 | UART DMA + IDLE 中断收字节 | 收到 `\r\n` 或空闲即触发一次帧回调 |
| 2 | 剥掉尾部 `\r`、`\n`、空白 | 循环从末尾丢，直到不是空白 |
| 3 | 长度检查 | 空帧丢弃；`> 30` 字节丢弃 |
| 4 | 匹配 `wake` | `strcmp(buf, "wake") == 0` → 唤醒事件 |
| 5 | 匹配 `cmd=` 前缀 | `strncmp(buf, "cmd=", 4) == 0` → 解析 id |
| 6 | 匹配 `done=` 前缀 | `strncmp(buf, "done=", 5) == 0` → 解析 id |
| 7 | id 解析 | 逐字符校验 0-9，累加十进制，范围外丢弃 |
| 8 | 都不匹配 | 静默忽略，不回错 |

## 四、注意事项 & 易错点

| # | 项目 | 说明 |
|:--:|------|------|
| 1 | `\n` 是单字节 0x0A | C 源码里 `"\n"` 编译后是 1 字节，**不是** `\`+`n` 两字节 |
| 2 | 别用 `printf` 发 UART | newlib 可能做 `\n` → `\r\n` 转换，破坏协议帧尾 |
| 3 | 用阻塞 `HAL_UART_Transmit` | 帧短（≤9 字节 @ 9600 ≈ 10ms），不需要异步 |
| 4 | 发送方向只带 `\n`，不带 `\r\n` | 协议规定 MCU→ASR 尾是 `\n`；带 `\r` 会被 ASRPRO 的 `trim_frame` 剥掉但浪费字节 |
| 5 | 接收方向必须支持 `\r\n` | ASRPRO 发过来的是 `\r\n`；只按 `\n` 分帧也行（`\r` 剥掉即可） |
| 6 | 帧缓冲 ≥ 32 字节 | 协议规定最大 32；建议 40-64 留余量 |
| 7 | `cmd` 和 `done` id 段不重叠 | cmd: 1-16；done: 17-41。用 id 就能反查是命令还是播报完成 |
| 8 | `wake` 帧没有 `=` 和 id | 别用 `parse_num` 去解，独立分支处理 |
| 9 | 未识别帧静默丢弃 | 不发错误码回 ASRPRO，避免协议膨胀 |
| 10 | id 数值上限 255 | 协议规定 dec ≤ 255；超出丢弃 |
| 11 | `done=<id>` 会**延后**到播完才到 | 别在 `play` 后立刻等 `done` 阻塞；改用状态机+超时 |
| 12 | `stop` 会取消 `done` | 主动 stop 后不会收到对应 done，业务逻辑要考虑 |
| 13 | UART 波特率两端严格 9600 | 8N1，无流控 |
| 14 | 上电时序 | ASRPRO 冷启动 ≈ 1-2 秒才能响应，MCU 早期发的 `play` 会丢 |
| 15 | 收到 `wake` 后 15 秒内要给命令 | ASRPRO 侧 `set_state_enter_wakeup(15000)` 超时会自动休眠 |
| 16 | 别在中断里做业务 | DMA/IDLE 中断只收帧入队列，业务在主循环消费 |
| 17 | 字符串比较用 `strcmp` 不用 `==` | C 里 `==` 比的是指针地址 |
| 18 | 帧长可变 | `play=1\n`(7) 到 `play=100\n`(9) 都合法；别写死长度 |
| 19 | 接收缓冲要清零 | 处理完立即置 `\0` 并复位长度，防脏数据残留 |
| 20 | 多帧粘包 | 理论上 ASRPRO 一次不会连发；但接收侧应能拆多帧（按 `\n` 循环切） |

## 五、状态机建议（业务层）

| 状态 | 触发 | 迁移 |
|------|------|------|
| IDLE | — | 收到 `wake` → LISTENING |
| LISTENING | 收到 `wake` | 收到 `cmd=X` → 执行命令回 IDLE；15 秒无 cmd → IDLE |
| PLAYING | 发出 `play=X` | 收到 `done=X` → IDLE；或主动 `stop` → IDLE；或超时 → IDLE |

## 六、串口调试小贴士

正点原子等串口助手调试 MCU 时：
- **勾选"发送新行"，输入框只打命令**（如 `play=17`），工具自动追加真正的 LF (0x0A)
- **不要**在输入框手打 `\n` 或 `\r\n`——那是两个字面字符（`\`+`n`），不是换行字节
- HEX 模式可直接手打 `70 6C 61 79 3D 31 37 0A` 验证 `play=17\n`
